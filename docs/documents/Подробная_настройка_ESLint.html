    Руководство по настройке ESLint

    Назначение: пошаговая инструкция для пользователя, который берёт шаблон ts-template-project и настраивает анализатор кода под свой проект. Все зависимости уже установлены; требуется скорректировать конфигурацию, пути, правила и интеграции.

    Шаблон использует ESLint 9 (Flat Config), ESM-конфиг и стек typescript-eslint. Конфигурация согласована с Prettier.

    1. Что вы получите

    • Единые правила качества кода для TypeScript и JavaScript.
    • Типо-осведомлённый анализ (recommendedTypeChecked, stylisticTypeChecked) там, где это нужно.
    • Корректный резолв алиасов из tsconfig (через ts-jest-resolver в тестах и явные маппинги при необходимости).
    • Совместимость с Prettier: форматирование и линтинг не конфликтуют.
    • Готовые интеграции для Jest (глобалы для тестов), lint-staged и CI.

    2. Какие файлы изменять

    Основные точки конфигурации:

    eslint.config.js        # основной ESM-конфиг (Flat Config)
    tsconfig.json           # IDE/разработка (парсер читает проект отсюда)
    tsconfig.base.json      # общие опции TypeScript, алиасы import
    package.json            # скрипты "lint", "lint:fix"

    3. Минимальный набор действий

    1. Уточните в eslint.config.js пути и области действия (files, раздел игноров).
    2. Проверьте, что parserOptions.project указывает на ваш tsconfig.json.
    3. (Опционально) Добавьте/исключите правила под корпоративные требования.
    4. Запустите проверку:
    npm run lint
    npm run lint:fix

    4. Рекомендуемая конфигурация eslint.config.js (Flat, ESM)

    Ниже пример, согласованный с шаблоном. Измените паттерны и правила по своему проекту. 

    // eslint.config.js (ESM, Flat Config)
    //
    // Назначение: единые правила анализа кода для JS/TS (NodeNext + ESM) с
    // типо-осведомленным режимом для исходников и тестов. Совместимо с Prettier.

    import js from '@eslint/js';
    import tseslint from 'typescript-eslint';
    import globals from 'globals';
    import prettier from 'eslint-config-prettier';
    import { fileURLToPath } from 'url';
    import { dirname } from 'path';

    const __filename = fileURLToPath(import.meta.url);
    const __dirname = dirname(__filename);

    export default [
        // 1) Глобальные исключения (Flat Config)
        {
            ignores: [
                'dist',
                'docs',
                'coverage',
                'site',
                'node_modules',
                '**/*.d.ts',
                '**/*.min.js',
            ],
        },

        // 2) JavaScript (без TS-плагина)
        {
            ...js.configs.recommended,
            files: ['**/*.{js,mjs,cjs}'],
            languageOptions: {
                ...js.configs.recommended.languageOptions,
                ecmaVersion: 2022,
                sourceType: 'module',
                globals: { ...globals.node },
            },
            rules: {
                ...js.configs.recommended.rules,
                // при необходимости: 'no-console': ['warn', { allow: ['warn', 'error'] }],
            },
        },

        // 3) TypeScript — базовые рекомендованные (без типов)
        ...tseslint.configs.recommended,

        // 4) TypeScript — рекомендуемые с типами
        ...tseslint.configs.recommendedTypeChecked.map((cfg) => ({
            ...cfg,
            files: ['src/**/*.{ts,tsx}', 'tests/**/*.{ts,tsx}'],
            languageOptions: {
                ...cfg.languageOptions,
                parserOptions: {
                    ...cfg.languageOptions?.parserOptions,
                    project: ['./tsconfig.json'],
                    tsconfigRootDir: __dirname,
                },
            },
        })),

        // 5) TypeScript — стилистические правила (type-checked, опционально)
        ...tseslint.configs.stylisticTypeChecked.map((cfg) => ({
            ...cfg,
            files: ['src/**/*.{ts,tsx}', 'tests/**/*.{ts,tsx}'],
            languageOptions: {
                ...cfg.languageOptions,
                parserOptions: {
                    ...cfg.languageOptions?.parserOptions,
                    project: ['./tsconfig.json'],
                    tsconfigRootDir: __dirname,
                },
            },
            // при необходимости:
            // rules: {
            //   '@typescript-eslint/consistent-type-imports': ['warn', { prefer: 'type-imports' }],
            //   '@typescript-eslint/no-confusing-void-expression': ['error', { ignoreArrowShorthand: true }],
            // },
        })),

        // 6) Тесты — глобалы Jest (без плагина)
        {
            files: ['tests/**/*.{ts,tsx,js,mjs,cjs}'],
            languageOptions: {
                globals: { ...globals.node, ...globals.jest },
            },
        },

        // 7) Общие опции линтера
        {
            linterOptions: {
                reportUnusedDisableDirectives: true,
            },
        },

        // 8) Интеграция с Prettier (должна быть последней)
        prettier,
    ];

    Что менять под свой проект:

    • Блок ignores — добавьте ваши каталоги сборки/кэша.
    • Блоки files — если исходники или тесты лежат в других путях.
    • parserOptions.project — путь(и) к актуальному tsconfig.json.
    • При необходимости задайте свои правила в секциях rules.

    5. Согласование с TypeScript

    • В tsconfig.base.json рекомендуется:

      • module: "NodeNext", moduleResolution: "NodeNext", moduleDetection: "force";
      • строгий режим (strict: true) и согласованные флаги;
      • алиасы в paths (например, "@/*": ["src/*"]).

    • В tsconfig.json (IDE):

      • включите "types": ["node", "jest"], если IDE не подхватывает их автоматически;
      • используйте noEmit: true и sourceMap: true для удобной разработки.

    6. Интеграция с Prettier

    Конфигурация уже подключает eslint-config-prettier последним элементом, чтобы отключить правила, конфликтующие с форматированием. Проверяйте, что:

    • в проекте есть prettier.config.js;
    • форматирование выполняется отдельно (npm run prettier) или через lint-staged.

    7. Интеграция с редактором (VS Code)

    Рекомендуемые настройки рабочего пространства:

    {
        "eslint.useFlatConfig": true,
        "eslint.workingDirectories": [{ "mode": "auto" }],
        "editor.codeActionsOnSave": {
            "source.fixAll.eslint": "explicit" // VS Code ≥1.75
        },
        "eslint.format.enable": false // форматирование делает Prettier
    }

    Если используете форматирование при сохранении, назначьте Prettier как default formatter и не смешивайте его с eslint --fix в одном шаге.

    8. Интеграция с lint-staged (pre-commit)

    Запускайте автоисправление только для staged-файлов. Пример: 

    // .lintstagedrc.mjs
    export default {
        ignore: ['dist/**', 'build/**', 'coverage/**', 'docs/**'],
        'src/**/*.{ts,tsx}': [
            'eslint --fix --cache --no-warn-ignored --max-warnings=0',
        ],
        'tests/**/*.{ts,tsx,js,cjs,mjs}': [
            'eslint --fix --cache --no-warn-ignored --max-warnings=0',
        ],
    };

    Хук:

    # .husky/pre-commit
    #!/usr/bin/env sh
    . "$(dirname -- "$0")/_/husky.sh"
    npx lint-staged

    9. Часто настраиваемые правила

    Ниже — типовые правила для библиотек. Добавляйте их в подходящий блок rules: 

    rules: {
      // согласованный стиль импортов типов
      '@typescript-eslint/consistent-type-imports': ['warn', { prefer: 'type-imports' }],

      // запрет неиспользуемых переменных с игнорированием _префикса
      '@typescript-eslint/no-unused-vars': ['warn', { argsIgnorePattern: '^_', varsIgnorePattern: '^_' }],

      // явные модификаторы доступа (на усмотрение команды)
      // '@typescript-eslint/explicit-member-accessibility': ['warn', { accessibility: 'no-public' }],

      // предпочтение nullish coalescing и optional chaining
      'no-restricted-syntax': [
        'warn',
        {
          selector: "BinaryExpression[operator='||'][right.value=null]",
          message: 'Рассмотрите ?? для null/undefined.'
        }
      ]
    }

    Уточняйте уровень строгости (off | warn | error) под ваш процесс.

    10. Скрипты в package.json

    В шаблоне уже определены:

    {
        "scripts": {
            "lint": "eslint \"src/**/*.ts\" \"tests/**/*.ts\"",
            "lint:fix": "eslint \"src/**/*.ts\" \"tests/**/*.ts\" --fix",
        },
    }

    При необходимости расширьте паттерны (*.js, конфиги) либо добавьте --cache для ускорения.

    11. CI и контроль в PR

    Рекомендуемый минимальный набор:

    npm ci
    npm run typecheck
    npm run lint
    npm test
    npm run build
    • Для больших репозиториев добавляйте --max-warnings=0, чтобы проваливать сборку при появлении предупреждений.

    12. Диагностика и типовые проблемы

    Parsing error: Cannot read file 'tsconfig.json' → Убедитесь, что parserOptions.project указывает на существующий tsconfig.json, а tsconfigRootDir равен корню репозитория (или каталогу конфигурации).

    ESLint couldn’t find the config "…" → Проверьте версии пакетов, корректность ESM-импорта и eslint.useFlatConfig: true в редакторе.

    Импорты с алиасом '@/…' не резолвятся → Убедитесь, что алиасы объявлены в tsconfig.base.json → paths. Для Jest используйте moduleNameMapper. Для ESLint (импорт-правил) можно дополнительно подключить eslint-import-resolver-typescript и плагин eslint-plugin-import (опционально).

    Конфликты форматирования → Проверьте, что блок prettier подключён последним, и Prettier запускается отдельно.

    13. Монорепозиторий

    • Держите базовый eslint.config.js в корне и добавляйте пакетные конфиги там, где логика различается (каждый — Flat Config).
    • В VS Code используйте "eslint.workingDirectories": [{ "mode": "auto" }].
    • Для проектных ссылок TypeScript включайте composite: true и актуализируйте parserOptions.project на каждый пакет.

    14. Контрольный список

    • [ ] Обновлён eslint.config.js: игноры, files, правила.
    • [ ] parserOptions.project указывает на ваш tsconfig.json.
    • [ ] Подключён prettier последним элементом.
    • [ ] Jest-глобалы добавлены для тестов.
    • [ ] Скрипты lint/lint:fix работают локально и в CI.
    • [ ] При необходимости расширены правила под командные стандарты.

    15. Пример тестовой связки

    tests/example.test.ts 

    import { sum } from '@/math/sum.js';

    describe('sum', () => {
        it('adds numbers', () => {
            expect(sum(2, 3)).toBe(5);
        });
    });

    src/math/sum.ts 

    export function sum(a: number, b: number): number {
        return a + b;
    }

    Настройка завершена. После корректировки конфигурации выполните npm run lint:fix, зафиксируйте изменения и добавьте проверку в CI.

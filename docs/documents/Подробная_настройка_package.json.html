    Руководство по настройке package.json

    Назначение: пошаговая инструкция для пользователя, который берет шаблон ts-template-project и настраивает метаданные, точки входа и сценарии публикации под свой проект. Все инструменты уже установлены; задача — скорректировать поля package.json, чтобы сборка, типы, публикация и документация работали предсказуемо.

    1. Цели и состав

    Цели настройки:

    • корректное имя пакета, описание, ссылки и лицензия;
    • согласование ESM-ориентации проекта с полями входа и exports;
    • предсказуемые пути к артефактам (dist/) и типам (.d.ts);
    • минимальная поверхность публикации (files), безопасная оптимизация (sideEffects);
    • удобные и повторяемые сценарии в scripts.

    Рекомендуемая структура:

    .
    ├─ src/                     # исходный код
    ├─ dist/                    # артефакты сборки (tsc)
    ├─ tests/                   # модульные тесты
    ├─ package.json             # конфигурация пакета (строгий JSON)
    └─ package.jsonc            # (необязательно) документированная копия для людей

    Реестры и менеджеры пакетов читают только package.json. Если вы хотите хранить комментированную версию, используйте рядом package.jsonc и синхронизируйте изменения.

    2. Минимальный перечень изменений

    1. Обновите идентификацию пакета: name, version, description, keywords, license, author, repository, homepage, bugs.
    2. Уточните целевую платформу: type: "module", engines.node, при необходимости packageManager.
    3. Подтвердите точки входа и типы: main, types, exports.
    4. Проверьте список публикуемых файлов: files.
    5. Отредактируйте scripts под ваш процесс (build, test, release).
    6. Убедитесь, что sideEffects корректно отражает поведение пакета.
    7. При публикации под scope проверьте publishConfig.access.

    3. Базовый шаблон package.json (ESM, библиотека)

    Ниже — ориентир. Замените подчёркнутые значения на свои. 

    {
        "name": "@your-scope/your-package",
        "version": "0.1.0",
        "description": "Краткое описание библиотеки (одна строка).",
        "keywords": ["typescript", "library", "esm"],
        "license": "MIT",

        "author": {
            "name": "Your Name",
            "email": "your@email",
            "url": "https://github.com/your",
        },

        "homepage": "https://github.com/your/your-repo#readme",
        "bugs": "https://github.com/your/your-repo/issues",
        "repository": {
            "type": "git",
            "url": "git+https://github.com/your/your-repo.git",
        },

        "type": "module",
        "engines": { "node": ">=18" },
        "packageManager": "npm@10", // или pnpm@x / yarn@x

        "sideEffects": false,
        "files": ["dist", "CHANGELOG.md", "README.md", "LICENSE"],

        "main": "./dist/index.js",
        "types": "./dist/index.d.ts",

        "exports": {
            ".": {
                "types": "./dist/index.d.ts",
                "import": "./dist/index.js",
                "default": "./dist/index.js",
            },
            "./package.json": "./package.json",
        },

        "scripts": {
            "prepublishOnly": "npm test && npm run build",
            "build": "npm run src:build && npm run docs:build",
            "clean": "npm run src:clean && npm run docs:clean",
            "src:clean": "rimraf dist .tsbuildinfo",
            "src:build": "npm run src:clean && tsc -p tsconfig.build.json && tsc-alias -p tsconfig.build.json",
            "docs:clean": "rimraf docs",
            "docs:build": "npm run docs:clean && typedoc",
            "docs:serve": "http-server docs -o -c-1",
            "test": "node --experimental-vm-modules ./node_modules/jest/bin/jest.js",
            "test:watch": "node --experimental-vm-modules ./node_modules/jest/bin/jest.js --watch",
            "typecheck": "tsc -p tsconfig.json --noEmit",
            "lint": "eslint \"src/**/*.ts\" \"tests/**/*.ts\"",
            "lint:fix": "eslint \"src/**/*.ts\" \"tests/**/*.ts\" --fix",
            "prettier": "prettier -w .",
            "release": "standard-version --config .versionrc.cjs",
            "release:patch": "standard-version --release-as patch --config .versionrc.cjs",
            "release:minor": "standard-version --release-as minor --config .versionrc.cjs",
            "release:major": "standard-version --release-as major --config .versionrc.cjs",
            "postrelease": "git push --follow-tags origin main",
            "prepare": "husky",
        },

        "publishConfig": { "access": "public" },
    }

    4. Ключевые поля и рекомендации

    4.1 Идентификация и ссылки

    • name: уникальное имя. Для публичных пакетов под организацией используйте scope: @org/pkg.
    • version: SemVer. Автоматизируйте повышением через standard-version.
    • description, keywords: краткие и точные. Улучшают поиск.
    • license: SPDX-идентификатор. Файл LICENSE должен быть в корне.
    • author: удобен для обратной связи; поля будут публичными.
    • homepage, bugs, repository: ссылки на документацию и трекер.

    4.2 Среда выполнения

    • type: "module": указывает на ESM по умолчанию. Для CJS используйте расширение .cjs.
    • engines.node: минимальная версия Node. Согласуйте с используемыми зависимостями.
    • packageManager: фиксирует менеджер и версию (полезно для CI и локальной синхронизации).

    4.3 Поверхность пакета и оптимизация

    • files: белый список публикуемого контента. Держите пакет минимальным.
    • sideEffects:
      • false — безопасно, если импорт модуля не вызывает глобальных изменений (регистрация полифилов, автосайд-эффекты и т. п.).
      • перечислите файлы/паттерны, если часть модулей действительно с побочными эффектами.

    4.4 Точки входа и типы

    • main: исторический вход для инструментов, не понимающих exports. Указывает на ESM-файл в dist.
    • types: декларации типов для редакторов и TypeScript.
    • exports: формирует публичную поверхность пакета. Рекомендуется всегда указывать корневой экспорт ".", а также явно экспортировать ./package.json, если его читают инструменты.

    Примеры подпутей (при необходимости):

    "exports": {
      ".": { "types": "./dist/index.d.ts", "import": "./dist/index.js" },
      "./utils": { "types": "./dist/utils/index.d.ts", "import": "./dist/utils/index.js" },
      "./package.json": "./package.json"
    }

    Для гибрида ESM/CJS потребуется отдельная сборка CJS и условные экспорты ("require"). Это выходит за рамки минимальной настройки.

    4.5 Сценарии (scripts)

    • Сборка: src:build использует tsc -p tsconfig.build.json, затем tsc-alias переписывает алиасы @/ в относительные пути.
    • Документация: typedoc собирает сайт документации в docs/.
    • Тесты: запуск Jest в ESM-режиме через --experimental-vm-modules.
    • Качество кода: lint, lint:fix, prettier.
    • Проверка типов: typecheck (без эмита).
    • Релизы: standard-version генерирует CHANGELOG и теги согласно conventional commits.
    • prepublishOnly: блокирует публикацию без зелёных тестов и сборки.
    • prepare: устанавливает Git-хуки Husky.

    5. package.jsonc (необязательно)

    Если вы храните рядом package.jsonc с комментариями:

    • следите за синхронизацией с package.json;
    • автоматизируйте вычитание комментариев при копировании (любой простой скрипт), чтобы не публиковать JSONC;
    • храните расширенные пояснения только в package.jsonc, package.json держите минимальным.

    6. Проверка перед публикацией

    1. Сборка:

      npm run build

    2. Набор файлов, которые уйдут в реестр:

      npm pack --dry-run

      Убедитесь, что в архив попали только нужные файлы из files.

    3. Предпросмотр экспорта типов (types/exports) — импортируйте пакет из временного проекта (или через pnpm link/npm link) и проверьте автодополнение.

    7. Частые варианты конфигурации

    7.1 Дополнительные поля, которые стоит рассмотреть

    • funding: ссылки на спонсорство.
    • sideEffects: массив шаблонов, если часть файлов вызывает побочные эффекты.
    • typesVersions: для сложных сценариев совместимости типов (обычно не требуется для простого ESM).

    7.2 Переопределение экспортов по подпутям

    Если в пакете несколько модулей верхнего уровня:

    "exports": {
      ".": { "types": "./dist/index.d.ts", "import": "./dist/index.js" },
      "./core": { "types": "./dist/core/index.d.ts", "import": "./dist/core/index.js" },
      "./builders": { "types": "./dist/builders/index.d.ts", "import": "./dist/builders/index.js" }
    }

    Не экспортируйте внутренние файлы, которые не должны быть частью публичного API.

    7.3 Публичная публикация под scope

    Скоуп-пакеты по умолчанию публикуются как restricted. Для публичного доступа: 

    "publishConfig": { "access": "public" }

    8. Проверка в CI

    Рекомендуется добавить шаги:

    npm ci
    npm run typecheck
    npm test
    npm run build
    npm run prettier:check
    npm pack --dry-run

    Это гарантирует повторяемость сборки и публикуемого состава.

    9. Контрольный список

    • [ ] Обновлены name, version, description, keywords, license, author.
    • [ ] Указаны homepage, bugs, repository.
    • [ ] Подтверждены type: "module" и engines.node.
    • [ ] Настроены main, types, exports и список files.
    • [ ] sideEffects отражает реальность (либо false).
    • [ ] Сценарии scripts соответствуют вашему процессу.
    • [ ] publishConfig.access корректен для вашего scope.
    • [ ] Проверка npm pack --dry-run показывает ожидаемый состав.
    • [ ] Выполнены сборка и тесты перед релизом.

    10. Пример финального package.json (краткий) 

    {
        "name": "@acme/awesome-lib",
        "version": "1.0.0",
        "description": "Небольшая библиотека для ...",
        "keywords": ["acme", "awesome", "typescript"],
        "license": "MIT",
        "author": { "name": "Acme", "url": "https://github.com/acme" },
        "homepage": "https://github.com/acme/awesome-lib#readme",
        "bugs": "https://github.com/acme/awesome-lib/issues",
        "repository": {
            "type": "git",
            "url": "git+https://github.com/acme/awesome-lib.git",
        },

        "type": "module",
        "engines": { "node": ">=18" },
        "packageManager": "npm@10",

        "sideEffects": false,
        "files": ["dist", "CHANGELOG.md", "README.md", "LICENSE"],

        "main": "./dist/index.js",
        "types": "./dist/index.d.ts",
        "exports": {
            ".": {
                "types": "./dist/index.d.ts",
                "import": "./dist/index.js",
                "default": "./dist/index.js",
            },
            "./package.json": "./package.json",
        },

        "scripts": {
            "prepare": "husky",
            "prepublishOnly": "npm test && npm run build",
            "build": "npm run src:build && npm run docs:build",
            "clean": "npm run src:clean && npm run docs:clean",
            "src:clean": "rimraf dist .tsbuildinfo",
            "src:build": "npm run src:clean && tsc -p tsconfig.build.json && tsc-alias -p tsconfig.build.json",
            "docs:clean": "rimraf docs",
            "docs:build": "npm run docs:clean && typedoc",
            "docs:serve": "http-server docs -o -c-1",
            "test": "node --experimental-vm-modules ./node_modules/jest/bin/jest.js",
            "test:watch": "node --experimental-vm-modules ./node_modules/jest/bin/jest.js --watch",
            "typecheck": "tsc -p tsconfig.json --noEmit",
            "lint": "eslint \"src/**/*.ts\" \"tests/**/*.ts\"",
            "lint:fix": "eslint \"src/**/*.ts\" \"tests/**/*.ts\" --fix",
            "prettier": "prettier -w .",
            "release": "standard-version --config .versionrc.cjs",
            "release:patch": "standard-version --release-as patch --config .versionrc.cjs",
            "release:minor": "standard-version --release-as minor --config .versionrc.cjs",
            "release:major": "standard-version --release-as major --config .versionrc.cjs",
            "postrelease": "git push --follow-tags origin main",
        },

        "publishConfig": { "access": "public" },
    }

    Настройка завершена. После обновления package.json выполните сборку, проверку составного архива через npm pack --dry-run и зафиксируйте изменения. Такой порядок обеспечивает предсказуемую публикацию и стабильное потребление библиотеки.

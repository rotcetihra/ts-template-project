    Руководство по настройке Prettier

    Назначение: пошаговая инструкция для пользователя, который берет шаблон ts-template-project и настраивает форматирование под свой проект. Все зависимости уже установлены; требуется лишь скорректировать конфигурации и, при необходимости, игноры и интеграции.

    1. Что вы получите

    • Единый стиль форматирования для исходников и текстовых файлов.
    • Предсказуемое форматирование при сохранении в редакторе, при коммите и в CI.
    • Минимум конфликтов с ESLint благодаря согласованным настройкам.

    2. Какие файлы изменять

    В репозитории шаблона уже есть готовые точки конфигурации. Пользователю обычно достаточно изменить следующие файлы: 

    prettier.config.js     # основные правила форматирования (ESM)
    .prettierignore        # пути/файлы, которые не форматируются
    .husky/pre-commit      # (опционально) хук, запускающий lint-staged
    .lintstagedrc.mjs      # (опционально) шаблоны файлов для pre-commit форматирования

    Скрипты prettier и prettier:check уже добавлены в package.json.

    3. Минимальный набор действий

    1. Откройте prettier.config.js и выберите стиль (см. раздел 4).

    2. Дополните .prettierignore своими путями сборки и бинарными ресурсами.

    3. При необходимости скорректируйте шаблоны в .lintstagedrc.mjs, чтобы форматировать только нужные типы файлов.

    4. В редакторе включите «форматировать при сохранении» (см. раздел 7).

    5. Выполните первичное выравнивание проекта:

      npm run prettier

      и зафиксируйте изменения.

    4. Опции, которые настраивают чаще всего

    Ниже перечислены параметры, которые обычно меняют при переносе шаблона в собственный проект. Для каждого параметра даны назначение и рекомендуемые варианты.

    4.1 Базовое форматирование

    Опция Назначение Типичные значения
    printWidth Целевая длина строки 80, 100, 120
    tabWidth Ширина отступа в пробелах 2 или 4
    useTabs Использовать табы вместо пробелов false (по умолчанию)
    semi Ставить ; в конце инструкций true
    singleQuote Одинарные кавычки в JS/TS true или false
    trailingComma Хвостовые запятые 'es5' или 'all'
    endOfLine Нормализация EOL 'lf'

    Рекомендации:

    • Библиотеки: printWidth: 100, trailingComma: 'all', singleQuote: true или false по команде.
    • Проекты с большим количеством Markdown: держите endOfLine: 'lf' и настроенный .gitattributes (см. раздел 9).

    4.2 Обрамление и парсер

    Опция Назначение Рекомендация
    bracketSpacing Пробелы в { a: 1 } true
    quoteProps Кавычки у ключей объектов 'as-needed'
    requirePragma Форматировать только файлы с @format false
    insertPragma Вставлять @format в начало false
    embeddedLanguageFormatting Форматировать встроенные блоки (JS в MD и т. п.) 'auto'

    4.3 JS/TS и разметка

    Опция Назначение Рекомендация
    arrowParens Скобки у единственного параметра стрелочных функций 'always'
    bracketSameLine Положение > у многострочных JSX/HTML тегов false
    jsxSingleQuote Одинарные кавычки в JSX-атрибутах false
    singleAttributePerLine По одному атрибуту на строке false
    htmlWhitespaceSensitivity Чувствительность к пробелам в HTML 'css'
    vueIndentScriptAndStyle Отступ внутри <script>/<style> в .vue false

    4.4 Markdown/MDX

    Опция Назначение Рекомендация
    proseWrap Перенос строк в Markdown 'preserve' или 'always'
    • Если у вас документация с длинными абзацами и важно минимизировать диффы — preserve.
    • Если предпочитаете «жесткую» ширину — задайте proseWrap: 'always' и выберите printWidth (например, 100).

    5. Рекомендуемый шаблон prettier.config.js

    Скопируйте и скорректируйте значения под свой стиль.

    // prettier.config.js
    export default {
        // 1) БАЗОВОЕ ФОРМАТИРОВАНИЕ
        printWidth: 100,
        tabWidth: 2,
        useTabs: false,
        semi: true,
        singleQuote: false,
        trailingComma: 'es5',
        bracketSpacing: true,
        endOfLine: 'lf',
        quoteProps: 'as-needed',

        // 2) ПАРСЕР И ОБЛАСТЬ ФОРМАТИРОВАНИЯ
        requirePragma: false,
        insertPragma: false,
        embeddedLanguageFormatting: 'auto',

        // 3) JS/TS
        arrowParens: 'always',

        // 4) JSX/HTML/Vue
        bracketSameLine: false,
        jsxSingleQuote: false,
        singleAttributePerLine: false,
        htmlWhitespaceSensitivity: 'css',
        vueIndentScriptAndStyle: false,

        // 5) MARKDOWN
        proseWrap: 'preserve',

        // 6) ПРИМЕР ПЕРЕОПРЕДЕЛЕНИЙ (по желанию)
        // overrides: [
        //   { files: ['**/*.md', '**/*.mdx'], options: { proseWrap: 'always' } },
        //   { files: ['**/*.{yml,yaml}'],     options: { singleQuote: false } }
        // ]
    };

    6. Игнорируемые пути: .prettierignore

    Расширьте список под структуру вашего проекта:

    node_modules/
    dist/
    build/
    coverage/
    docs/
    .eslintcache
    *.min.js
    *.map

    # Бинарные и медиа
    **/*.{png,jpg,jpeg,gif,webp,svg,ico,pdf}

    Prettier не использует .gitignore автоматически. Дублируйте ключевые исключения.

    7. Интеграция с редактором (VS Code)

    Включите локальный Prettier и форматирование при сохранении: 

    {
        "editor.defaultFormatter": "esbenp.prettier-vscode",
        "editor.formatOnSave": true,
        "prettier.prettierPath": "./node_modules/prettier",
        "prettier.configPath": "./prettier.config.js"
    }

    При работе с несколькими форматтерами используйте профили по языкам через блоки [language].

    8. Интеграция с ESLint и lint-staged

    8.1 ESLint

    В шаблоне уже подключен eslint-config-prettier. При необходимости проверьте в конфиге ESLint наличие строки: 

    "extends": ["prettier"]

    Это отключает правила ESLint, дублирующие форматирование.

    8.2 lint-staged (pre-commit)

    В .lintstagedrc.mjs отрегулируйте паттерны под свой проект, например добавьте или уберите типы файлов: 

    export default {
        ignore: ['dist/**', 'build/**', 'coverage/**', 'docs/**'],
        '**/*.{js,jsx,ts,tsx,json,md,yml,yaml}': [
            'prettier -w --log-level warn --cache',
        ],
    };

    9. Рекомендации для командной работы и CI

    1. Единый EOL для всех участников: 

      # .gitattributes
      * text=auto eol=lf

    2. Проверка в CI: используйте режим проверки без записи: 

      npm run prettier:check

    3. Первичное выравнивание: первый запуск prettier -w . часто даёт большой дифф. Сделайте отдельный коммит «chore: format with Prettier».

    10. Частые варианты и советы

    • Больше кода на одной строке: увеличьте printWidth до 100/120.
    • Единый стиль кавычек: выберите singleQuote: true или false и придерживайтесь его.
    • Минимум диффов в списках: trailingComma: 'all'.
    • Документационные репозитории: proseWrap: 'always' и printWidth: 100.
    • Monorepo: держите один корневой конфиг и при необходимости используйте overrides для пакетов.

    11. Проверка результата

    • Локально:

      npm run prettier:check

    • Форматирование:

      npm run prettier

    • Pre-commit: выполните тестовый коммит измененного файла и убедитесь, что форматирование применяется автоматически.

    12. Контрольный список

    • [ ] Обновлен prettier.config.js (базовые опции под проект).
    • [ ] Заполнен .prettierignore проектными путями.
    • [ ] В редакторе включено форматирование при сохранении.
    • [ ] В ESLint подключен eslint-config-prettier.
    • [ ] В lint-staged заданы нужные паттерны.
    • [ ] В CI добавлена проверка prettier -c ..
    • [ ] Выполнено первичное форматирование кода.

    Готово. После этих шагов Prettier будет работать одинаково в редакторе, при коммите и в CI, а пользователю останется лишь поддерживать единый стиль в рамках выбранных опций.

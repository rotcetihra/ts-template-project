    Руководство по настройке standard-version

    Назначение: пошаговая инструкция по выпуску версий и ведению CHANGELOG в библиотеке на базе ts-template-project. В шаблоне все зависимости уже установлены; требуется уточнить конфигурацию, убедиться в корректности сообщений коммитов и настроить рабочий процесс релиза.

    Что делает standard-version

    Инструмент автоматизирует выпуск версий на основе Conventional Commits:

    • вычисляет новый номер версии по истории коммитов (SemVer);
    • обновляет package.json и при необходимости lock-файлы;
    • формирует или дополняет CHANGELOG.md;
    • создает релизный коммит и git-тег.

    Публикацию в реестр (npm) standard-version не выполняет. Публиковать вы будете отдельной командой.

    Предварительные требования

    1. Conventional Commits. Сообщения коммитов должны соответствовать схеме type(scope): subject. В шаблоне уже настроен Commitlint (см. соответствующее руководство).

    2. Чистый рабочий каталог. Перед релизом не должно быть незакоммиченных изменений и незапушенных тегов.

    3. Актуальные метаданные пакета. В package.json должны быть корректные поля name, version, repository.

    Структура файлов и скриптов

    Минимальный набор в шаблоне:

    .versionrc.cjs             # конфигурация standard-version
    package.json               # скрипты релиза
    CHANGELOG.md               # будет создан/обновлён автоматически

    Скрипты (уже присутствуют):

    {
        "scripts": {
            "release": "standard-version --config .versionrc.cjs",
            "release:patch": "standard-version --release-as patch --config .versionrc.cjs",
            "release:minor": "standard-version --release-as minor --config .versionrc.cjs",
            "release:major": "standard-version --release-as major --config .versionrc.cjs",
            "postrelease": "git push --follow-tags origin main",
        },
    }

    При необходимости замените main на вашу основную ветку.

    Базовая конфигурация .versionrc.cjs

    Шаблон уже содержит улучшенный вариант. Ниже — опорный пример (заменять не обязательно, если вы уже используете его): 

    // .versionrc.cjs
    /** @type {import('standard-version').Options} */
    module.exports = {
        infile: 'CHANGELOG.md',
        header: '# Changelog\n\nВсе заметные изменения в этом проекте задокументированы в этом файле.\n',
        tagPrefix: 'v',
        preset: 'conventionalcommits',

        packageFiles: [{ filename: 'package.json', type: 'json' }],
        bumpFiles: [
            { filename: 'package.json', type: 'json' },
            { filename: 'package-lock.json', type: 'json' },
        ],

        releaseCommitMessageFormat:
            'chore(release): {{currentTag}}\n\n{{#if notes}}{{notes}}{{/if}}',

        types: [
            { type: 'feat', section: 'Новые возможности', hidden: false },
            { type: 'fix', section: 'Исправления ошибок', hidden: false },
            { type: 'perf', section: 'Производительность', hidden: false },
            { type: 'security', section: 'Безопасность', hidden: false },
            { type: 'deps', section: 'Зависимости', hidden: false },

            { type: 'refactor', section: 'Рефакторинг', hidden: true },
            { type: 'docs', section: 'Документация', hidden: true },
            { type: 'test', section: 'Тесты', hidden: true },
            { type: 'build', section: 'Система сборки', hidden: true },
            { type: 'ci', section: 'CI', hidden: true },
            { type: 'chore', section: 'Обслуживание', hidden: true },
            { type: 'revert', section: 'Откаты', hidden: true },
            { type: 'style', section: 'Стиль кода', hidden: true },
        ],

        // Явные шаблоны ссылок (опционально)
        commitUrlFormat:
            'https://github.com/rotcetihra/ts-template-project/commit/{{hash}}',
        compareUrlFormat:
            'https://github.com/rotcetihra/ts-template-project/compare/{{previousTag}}...{{currentTag}}',
        issueUrlFormat:
            'https://github.com/rotcetihra/ts-template-project/issues/{{id}}',
        userUrlFormat: 'https://github.com/{{user}}',

        // Дополнительные флаги, если понадобятся:
        // sign: true,          // подписывать теги (git tag -s)
        // commitAll: true,     // коммитить все изменённые файлы
        // dryRun: true,        // прогон без изменений в репозитории
        skip: { changelog: false, tag: false, commit: false },
    };

    Что важно адаптировать под себя:

    • адреса GitHub в *UrlFormat или просто удалите их и используйте автоопределение по repository в package.json;
    • список types и видимость разделов;
    • ветку в скрипте postrelease.

    Ежедневный цикл релиза

    1. Убедитесь, что тесты и сборка проходят локально: 

      npm test && npm run build

    2. Выберите тип релиза и выполните один из скриптов: 

      npm run release:patch  # исправления
      npm run release:minor  # новые возможности без breaking changes
      npm run release:major  # есть breaking changes
      # либо полуавтоматически:
      npm run release        # тип определяется по истории коммитов

    3. Просмотрите изменения (package.json, CHANGELOG.md, созданный тег).

    4. Запушьте коммит и теги:

      npm run postrelease

    5. Опубликуйте пакет (если требуется):

      npm publish --access public

    В шаблоне присутствует prepublishOnly: перед публикацией будут запущены тесты и сборка.

    Выбор типа релиза (patch/minor/major)

    • standard-version может сам определить тип релиза по истории:

      • feat: обычно означает minor;
      • fix: означает patch;
      • наличие BREAKING CHANGE: в body/footer — major.

    • Если хотите жёстко задать тип, используйте флаг --release-as или готовый скрипт (release:minor и т. п.).

    Предрелизы (alpha/beta/rc)

    Примеры:

    # первый предрелиз: 1.2.3 -> 1.2.4-alpha.0
    npx standard-version --prerelease alpha --config .versionrc.cjs

    # следующий предрелиз: 1.2.4-alpha.0 -> 1.2.4-alpha.1
    npx standard-version --prerelease alpha --config .versionrc.cjs

    # вручную задать тип + предрелиз
    npx standard-version --release-as minor --prerelease rc --config .versionrc.cjs

    После завершения цикла предрелизов выполните обычный релиз (без --prerelease), чтобы получить стабильную версию.

    Настройка разделов CHANGELOG

    Сопоставление типов коммитов разделам задаётся в поле types .versionrc.cjs. Рекомендации:

    • оставляйте видимыми то, что интересно пользователям пакета (feat, fix, perf, security, deps);
    • «внутренние» типы (chore, ci, build, style, test, refactor) можно скрыть через hidden: true.

    Ссылки на коммиты/тикеты и репозиторий

    Если в package.json корректно заполнено поле repository, standard-version автоматически проставит ссылки в CHANGELOG. Явные шаблоны (commitUrlFormat, compareUrlFormat, issueUrlFormat, userUrlFormat) пригодятся, если автоопределение не сработало или вы используете нестандартный хостинг.

    Интеграция с CI

    Минимальный конвейер:

    npm ci
    npm run typecheck
    npm test
    npm run build
    npm run release -- --dry-run     # в PR: проверка того, что будет сгенерировано

    В защищённых ветках запускайте релизные скрипты вручную (или отдельной джобой), затем git push --follow-tags и публикация.

    Диагностика и типовые проблемы

    Нет предыдущих тегов, релиз не стартует Используйте флаг --first-release для первичного выпуска.

    Неправильная версия вычисляется автоматически Укажите тип явно (--release-as patch|minor|major) или убедитесь, что в коммитах верно проставлены feat/fix/BREAKING CHANGE:.

    Некорректные ссылки в CHANGELOG Проверьте поле repository в package.json и/или явно задайте шаблоны URL в .versionrc.cjs.

    Не обновляется lock-файл Добавьте соответствующий файл в bumpFiles (package-lock.json, pnpm-lock.yaml, yarn.lock).

    Нужно пропустить шаг Используйте skip в конфиге, например: 

    skip: { changelog: false, tag: false, commit: false }

    Контрольный список

    • [ ] Сообщения коммитов соответствуют Conventional Commits (Commitlint настроен).
    • [ ] .versionrc.cjs заполнен и отражает ваш процесс релиза.
    • [ ] Скрипты release:* в package.json указывают на .versionrc.cjs.
    • [ ] В bumpFiles перечислены все файлы с версией, которые надо обновлять.
    • [ ] Ветки и postrelease соответствуют вашему репозиторию.
    • [ ] Процесс публикации (npm publish) согласован с вашей стратегией релизов.

    После этой настройки выпуск новой версии сводится к одной команде, а CHANGELOG формируется предсказуемо и без ручного редактирования.

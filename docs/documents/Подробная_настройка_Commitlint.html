    Руководство по настройке Commitlint

    Назначение: пошаговая инструкция для пользователя, который берет шаблон ts-template-project и настраивает проверку сообщений коммитов в стиле Conventional Commits. Все зависимости уже установлены; требуется лишь скорректировать конфигурацию, добавить/изменить типы, области и подключить хук commit-msg.

    1. Что вы получите

    • Единый стиль сообщений коммитов, совместимый с автогенерацией CHANGELOG и релизов.
    • Строгую валидацию структуры: type(scope): subject + body/footer.
    • Гибкую настройку правил: допустимые типы, обязательность области, длина заголовка, регистр темы и др.
    • Интеграцию с Husky и подготовку к релизному пайплайну (standard-version).

    2. Какие файлы изменять

    Типовой набор:

    commitlint.config.cjs    # конфигурация Commitlint (CommonJS)
    .husky/commit-msg        # git-хук для проверки сообщений
    package.json             # скрипты релизов (уже настроены в шаблоне)

    В шаблоне уже стоят пакеты @commitlint/cli и @commitlint/config-conventional.

    3. Минимальный набор действий

    1. Создайте/отредактируйте commitlint.config.cjs (см. раздел 4).
    2. Подключите хук commit-msg (см. раздел 5).
    3. Согласуйте типы и, при необходимости, области (scope) под домены вашего проекта.
    4. Проверьте локально: выполните тестовый коммит и убедитесь, что некорректные сообщения блокируются.

    4. Рекомендуемая конфигурация commitlint.config.cjs

    Ниже конфигурация, согласованная с шаблоном. Измените типы/правила под свои требования. 

    // commitlint.config.cjs

    /** @type {import('@commitlint/types').UserConfig} */
    module.exports = {
        // Базовый набор правил Conventional Commits
        extends: ['@commitlint/config-conventional'],

        // Дополнительные правила проекта
        rules: {
            // HEADER
            'header-max-length': [2, 'always', 100], // ограничение длины заголовка
            'type-empty': [2, 'never'], // тип обязателен
            'type-enum': [
                // допустимые типы
                2,
                'always',
                [
                    'build', // сборка, инструменты, CI/CD
                    'chore', // прочие задачи без изменения кода/тестов
                    'ci', // конфигурации CI
                    'docs', // документация
                    'feat', // новая функциональность
                    'fix', // исправление дефекта
                    'perf', // производительность
                    'refactor', // рефакторинг без изменения поведения
                    'revert', // откат
                    'style', // форматирование, пробелы, точки с запятой и т. п.
                    'test', // тесты
                    'release', // релизные коммиты (теги, версии)
                    'security', // устранение уязвимостей
                    'deps', // обновления зависимостей
                ],
            ],
            'subject-empty': [2, 'never'], // тема обязательна
            'subject-case': [2, 'always', ['lower-case']], // тема в нижнем регистре
            'subject-full-stop': [2, 'never', '.'], // без точки в конце заголовка

            // SCOPE
            'scope-empty': [0], // область не обязательна
            'scope-case': [2, 'always', 'kebab-case'], // формат области

            // BODY / FOOTER
            'body-leading-blank': [2, 'always'], // пустая строка перед body
            'footer-leading-blank': [2, 'always'], // пустая строка перед footer
            // Примеры дополнительных ограничений:
            // 'body-max-line-length': [2, 'always', 120],
            // 'footer-max-line-length': [2, 'always', 120],
            // 'references-empty': [2, 'never'], // требовать ссылки на задачи
        },

        // (Опционально) игнорировать некоторые сообщения
        // ignores: [(message) => message.startsWith('WIP')],
    };

    Что менять под свой проект:

    • type-enum: добавляйте/удаляйте типы, которые реально используете.
    • scope-case и scope-empty: зафиксируйте стиль областей и их обязательность.
    • subject-case: если требуется свободный регистр (например, с прописной буквы для русского), отключите правило ([0]).
    • Лимиты длины: подберите комфортный header-max-length и ограничения для body/footer.

    5. Подключение Git-хука commit-msg (Husky)

    Создайте хук, который запускает валидацию на каждый коммит: 

    # .husky/commit-msg
    #!/usr/bin/env sh
    . "$(dirname -- "$0")/_/husky.sh"

    npx --no-install commitlint --edit "$1"

    Права на исполнение:

    chmod +x .husky/commit-msg

    В шаблоне скрипт prepare: husky уже есть. Если Husky не установлен, выполните npx husky init в корне.

    6. Правила Conventional Commits: краткий справочник

    6.1 Заголовок

    Формат:

    <type>(<scope>)?: <subject>

    Примеры:

    feat(core): добавить фабрику билдеров
    fix(utils): корректный парсинг JSON ошибок
    docs: обновить раздел "Быстрый старт"

    Рекомендации:

    • Type отражает природу изменения (см. type-enum).
    • Scope описывает подсистему, подпакет или слой: core, builders, utils, types, docs.
    • Subject в императивной форме, без точки и в нижнем регистре.

    6.2 Body

    • Подробности изменения, мотивация, риски, способы тестирования.
    • Каждую логическую мысль — отдельным абзацем.
    • Перед body — пустая строка.

    6.3 Footer

    • Ссылки на задачи/тикеты, пометки BREAKING CHANGE: и др.

    • Примеры:

      BREAKING CHANGE: переименовано поле options.timeout -> requestTimeout
      Closes #123
      Ref #456

    6.4 Семантические версии

    • feat: — обычно MINOR;
    • fix: — PATCH;
    • BREAKING CHANGE: в body/footer — MAJOR.
    • Совместимо с инструментами релизов (см. раздел 8).

    7. Рекомендации по областям (scope)

    Заранее договоритесь о перечне областей, которые отражают архитектуру проекта. Примеры для шаблона: 

    core, builders, utils, types, docs, ci, build, release, tests

    Стилистика: kebab-case. Если область не нужна, не указывайте её вовсе.

    8. Интеграция с релизами (standard-version)

    В шаблоне уже настроены скрипты релизов:

    • release:patch / release:minor / release:major
    • release (по конфигу .versionrc.cjs)

    Commitlint помогает поддерживать корректные типы и BREAKING CHANGE, чтобы standard-version формировал CHANGELOG предсказуемо.

    9. Интерактивные подсказки (опционально)

    Если хочется интерактивного выбора type/scope/subject, рассмотрите установку промпта: 

    npm i -D @commitlint/prompt-cli

    Запуск:

    npx commit

    И (по желанию) настройте секцию prompt в конфиге, чтобы подсказать доступные scopes.

    10. CI и контроль в PR

    Добавьте шаг, который валидирует последний коммит или диапазон: 

    # проверить сообщение HEAD
    npx commitlint --from HEAD~1 --to HEAD

    Для пул-реквестов в CI можно валидировать все коммиты ветки: 

    npx commitlint --from origin/main --to HEAD

    11. Диагностика и типичные проблемы

    Сообщение заблокировано, но формат верный визуально Проверьте пробел после двоеточия и отсутствие точки в конце заголовка, а также регистр темы.

    Слишком длинный заголовок Сократите формулировку или перенесите детализацию в body. Заголовок — до 100 символов (настраивается).

    Нужен тип, отсутствующий в списке Добавьте его в type-enum. Не злоупотребляйте: держите список коротким и понятным команде.

    Русскоязычные заголовки с прописной буквы Отключите/смягчите subject-case: 

    'subject-case': [0]

    12. Контрольный список

    • [ ] Создан/обновлен commitlint.config.cjs с типами и правилами.
    • [ ] Подключен хук .husky/commit-msg.
    • [ ] Определены области (scope) и стиль их записи.
    • [ ] Включены правила про пустые строки перед body/footer.
    • [ ] Настроен пайплайн релизов (standard-version) и CHANGELOG.

    13. Примеры корректных сообщений 

    feat(core): добавить фабрику клиентов http

    описание решения, мотивация и ограничения
    refactor: выделение общего конфига будет выполнено отдельно

    Closes #231

    fix(utils): корректная обработка таймаута при повторной попытке

    причина дефекта и способ воспроизведения
    добавлены юнит-тесты для edge-case

    Ref #412

    perf(builders): снизить кол-во аллокаций в parseConfig

    BREAKING CHANGE: options.timeout переименован в requestTimeout

    После этих шагов все коммиты будут единообразными, читаемыми и пригодными для автоматического формирования версий и журналов изменений.

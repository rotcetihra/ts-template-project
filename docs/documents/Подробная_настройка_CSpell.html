    Руководство по настройке CSpell

    Назначение: пошаговая инструкция для пользователя, который берет шаблон ts-template-project и настраивает проверку орфографии под свой проект. Все зависимости уже установлены; требуется скорректировать конфигурацию, игноры, пользовательские словари и интеграцию с lint-staged/CI.

    1. Что вы получите

    • Проверку орфографии в коде, документации и конфигурационных файлах.
    • Поддержку нескольких локалей (например, en и ru) и внешних словарей.
    • Низкий «шум» за счет корректных игноров и регулярных выражений.
    • Интеграцию с pre-commit (lint-staged) и CI.

    2. Какие файлы изменять

    Типовая структура:

    .
    ├─ cspell.json                # основная конфигурация (JSON/JSONC)
    ├─ .cspell/
    │  └─ words.txt       # (опционально) ваш проектный словарь
    ├─ .lintstagedrc.mjs          # (опционально) pre-commit проверка
    └─ package.json               # скрипты, если нужны

    В шаблоне конфиг уже есть. Обычно вам нужно скорректировать локали (language), игноры (ignorePaths) и пользовательские слова (words) либо вынести их в отдельный файл-словарь.

    3. Минимальный набор действий

    1. Откройте cspell.json и установите нужные локали, например "language": "en,ru".

    2. Убедитесь, что импортирован внешний словарь(и) для ваших языков (напр., @cspell/dict-ru_ru).

    3. Дополните ignorePaths путями сборки и бинарных ресурсов вашего проекта.

    4. Перенесите повторяющиеся термины в words или в отдельный словарь .cspell/words.txt и подключите его (см. раздел 6).

    5. Проверьте работу локально:

      npx cspell lint --no-must-find-files --gitignore --unique --color --quiet --no-progress .

    4. Рекомендуемая конфигурация cspell.json (сгруппировано и с комментариями)

    Ниже приведен вариант конфигурации на основе вашего файла, дополненный типичными практиками. Формат — JSONC (с комментариями). 

    {
        // =====================================================================
        // 1) БАЗОВЫЕ ПАРАМЕТРЫ
        // =====================================================================

        // Версия формата конфигурации CSpell.
        "version": "0.2",

        // Список локалей через запятую. Включаются соответствующие словари.
        // Примеры: "en", "en-GB", "en-US", "ru", или комбинированно: "en,ru".
        "language": "en,ru",

        // Импорт внешних словарей. Требует соответствующих пакетов в devDependencies.
        // Для русского языка — @cspell/dict-ru_ru.
        "import": ["@cspell/dict-ru_ru/cspell-ext.json"],

        // По желанию: учитывать .gitignore по умолчанию.
        // Эквивалентно флагу CLI --gitignore.
        "useGitignore": true,

        // =====================================================================
        // 2) ОБЛАСТЬ ПРОВЕРКИ И ИГНОРЫ
        // =====================================================================

        // Пути/паттерны, которые следует исключить из проверки орфографии.
        "ignorePaths": [
            "node_modules",
            "dist",
            "build",
            "coverage",
            "docs",
            "site",
            ".husky",
            ".github",
            // Часто встречающиеся кэши и артефакты
            ".turbo",
            ".next",
            "*.min.js",
            "**/*.d.ts",
            "**/*.map",
        ],

        // Регулярные выражения для игнорирования фрагментов текста в проверяемых файлах.
        "ignoreRegExpList": [
            // URL (http/https)
            "https?://\\S+",
            // UUID и хэши (7+ шестнадцатеричных символов)
            "\\b[0-9a-fA-F]{7,}\\b",
            // Семантические версии (например, 1.2.3, 1.2.3-beta.1)
            "\\b\\d+\\.\\d+\\.\\d+(?:-[0-9A-Za-z.-]+)?\\b",
            // Блоки кода в Markdown (```...```)
            "```[\\s\\S]*?```",
        ],

        // =====================================================================
        // 3) СЛОВАРИ И ЯЗЫКОВЫЕ НАСТРОЙКИ
        // =====================================================================

        // Определения собственных словарей (локальные файлы со словами, по одному в строке).
        // Создайте .cspell/words.txt и храните там проектные термины.
        "dictionaryDefinitions": [
            {
                "name": "words",
                "path": ".cspell/words.txt",
                "addWords": true,
            },
        ],

        // Активируемые словари (из import и из dictionaryDefinitions).
        "dictionaries": ["words"],

        // Тонкая настройка для отдельных типов файлов.
        "languageSettings": [
            {
                // Markdown и MDX: разрешаем составные слова и снижаем шум
                "languageId": ["markdown", "mdx"],
                "allowCompoundWords": true,
            },
            {
                // JSON/JSONC/YAML: чаще встречаются идентификаторы и ключи конфигов
                "languageId": ["json", "jsonc", "yaml", "yml"],
                "caseSensitive": false,
            },
        ],

        // =====================================================================
        // 4) ПОЛЬЗОВАТЕЛЬСКИЕ СЛОВА И ОГРАНИЧЕНИЯ
        // =====================================================================

        // Пользовательские слова, признаваемые корректными (имена собственные, термины).
        // Перенесите сюда стабильные термины, либо храните их в words.txt.
        "words": ["Arhitector", "rotcetihra"],

        // Слова, которые следует считать ошибочными (запрещенные варианты).
        // Удобно для фиксации частых опечаток в команде.
        "flagWords": ["teh", "recieve", "adress"],
    }

    Если words.txt отсутствует — создайте файл и добавляйте в него по одному слову на строку. Это удобнее, чем постоянно расширять массив words в JSON.

    5. Снижение «шума» в отчётах

    • Поддерживайте актуальный список игноров: артефакты сборки (dist, coverage, *.map), кэши (.turbo, .next), типовые бинарные расширения.

    • Используйте ignoreRegExpList для:

      • URL, версий SemVer, UUID/хэшей, больших блоков кода в Markdown.

    • Для Markdown включите allowCompoundWords, чтобы не ругаться на составные термины.

    6. Пользовательские словари

    Долгий список терминов лучше хранить в отдельном файле: 

    .cspell/words.txt

    Подключение уже показано в dictionaryDefinitions/dictionaries. Это облегчает ревью и поддержку.

    7. Интеграция с lint-staged (pre-commit)

    Пример: проверять только изменённые Markdown/текстовые файлы. 

    // .lintstagedrc.mjs
    export default {
        '**/*.{md,mdx,txt}': [
            'cspell lint --no-must-find-files --gitignore --unique --color --quiet --no-progress',
        ],
        '**/*.{json,jsonc,yml,yaml}': [
            'cspell lint --no-must-find-files --gitignore --unique --color --quiet --no-progress',
        ],
    };

    Для исходников, где уже есть строгие линтеры, имеет смысл запускать CSpell только на документации и конфигурациях, чтобы не дублировать шум.

    8. Интеграция с редактором (VS Code)

    Установите расширение Code Spell Checker и добавьте настройки рабочего пространства: 

    {
        "cSpell.configFile": "./cspell.json",
        "cSpell.language": "en,ru",
        "cSpell.useGitignore": true,
        "cSpell.enabledLanguageIds": [
            "markdown",
            "mdx",
            "json",
            "jsonc",
            "yaml",
            "yml",
            "javascript",
            "typescript"
        ]
    }

    9. Интеграция с CI

    Минимальный шаг в конвейере:

    npx cspell lint --no-must-find-files --gitignore --unique --color --no-progress .
    • --no-must-find-files — не считать ошибкой отсутствие совпавших файлов.
    • --gitignore — учитывать .gitignore.
    • --unique — не дублировать одно и то же слово в выводе.
    • --no-progress — компактный вывод для CI.

    10. Рабочий процесс добавления слов

    1. Запустите CSpell локально или в PR.
    2. Если слово корректное, добавьте его в .cspell/words.txt или в words в cspell.json.
    3. Если слово — частая опечатка, добавьте в flagWords.
    4. Если это шум (URL/хэш/версия), добавьте регулярное выражение в ignoreRegExpList.

    11. Монорепозиторий

    • Храните корневой cspell.json и подключайте пакетные словари через overrides или отдельные dictionaryDefinitions.
    • Можно завести отдельные words.txt на пакет и подключать их соответствующими глоссариями.

    12. Контрольный список

    • [ ] Указаны локали и импортированы нужные внешние словари.
    • [ ] Настроены ignorePaths и ignoreRegExpList.
    • [ ] Проектные термины вынесены в words или .cspell/words.txt.
    • [ ] Интеграция с lint-staged и/или CI настроена.
    • [ ] Проверка выполняется локально и в PR.

    После этих шагов CSpell будет работать предсказуемо: фиксировать реальные опечатки, не создавать лишнего «шума» и поддерживать общий словарь команды.

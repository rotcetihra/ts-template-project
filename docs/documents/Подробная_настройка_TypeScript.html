    Руководство по настройке tsconfig

    Назначение: пошаговая инструкция по настройке TypeScript-конфигураций для библиотеки на базе преднастроенной среды ts-template-project. Руководство покрывает три уровня конфигурации: базовую (tsconfig.base.json), сборочную (tsconfig.build.json) и для разработки/IDE (tsconfig.json). Все примеры даны в формате JSONC с комментариями.

    1. Цели и структура

    Цели конфигурации:

    • единая строгая база правил и таргета JavaScript;
    • корректная поддержка ESM/NodeNext, карт исходников и деклараций типов;
    • согласование путей импорта (aliases) с реальной структурой проекта;
    • предсказуемая интеграция с инструментами (TypeDoc, тесты, IDE, CI).

    Рекомендуемая структура (по умолчанию в шаблоне):

    .
    ├─ src/                   # исходный код библиотеки
    ├─ tests/                 # модульные тесты
    ├─ dist/                  # результирующие артефакты сборки
    ├─ tsconfig.base.json     # базовые правила и целевая платформа
    ├─ tsconfig.build.json    # конфигурация сборки (d.ts, sourcemap и т.д.)
    └─ tsconfig.json          # конфигурация для IDE/разработки (noEmit)

    2. Базовая конфигурация: tsconfig.base.json

    Назначение: единая точка, где задаются таргет, модель модулей, строгие проверки и алиасы импортов. На неё ссылаются производные конфигурации.

    Рекомендуемый вариант:

    // tsconfig.base.json
    {
        "compilerOptions": {
            // ------------------------ ЯЗЫК И СРЕДА -------------------------------
            // Современный целевой стандарт. Для Node 18+ и современных браузеров
            // ES2022 обеспечивает оптимальный набор возможностей.
            "target": "ES2022",
            "lib": ["ES2022"],

            // Глобальные типы для Node и тестов (при необходимости).
            "types": ["node", "jest"],

            // --------------------------- МОДУЛИ -----------------------------------
            // Режим модулей NodeNext (ESM-first) и соответствующее разрешение модулей.
            "module": "NodeNext",
            "moduleResolution": "NodeNext",

            // Все файлы рассматриваются как модули (не скрипты).
            "moduleDetection": "force",

            // ------------------------ СТРОГИЕ ПРОВЕРКИ ----------------------------
            "strict": true,
            "exactOptionalPropertyTypes": true,
            "noUncheckedIndexedAccess": true,
            "useUnknownInCatchVariables": true,
            "noImplicitOverride": true,
            "noPropertyAccessFromIndexSignature": true,

            // --------------------- КОМПИЛЯЦИЯ И ВЫВОД -----------------------------
            "isolatedModules": true,
            "skipLibCheck": true,
            "esModuleInterop": true,
            "resolveJsonModule": true,
            "verbatimModuleSyntax": true,
            "useDefineForClassFields": true,
            "forceConsistentCasingInFileNames": true,

            // --------------------- ПУТИ И АЛИАСЫ ----------------------------------
            "baseUrl": ".",
            "paths": {
                "@/*": ["src/*"],
                "tests/*": ["tests/*"],
            },
        },
    }

    Пояснения и вариации:

    • Если библиотека предназначена для браузера, добавьте DOM-типы: lib: ["ES2022", "DOM"].
    • Если вы не используете Jest, уберите "jest" из types.
    • verbatimModuleSyntax: true фиксирует исходные import/export без переписывания, что помогает при ESM.
    • Для поддержания совместимости с более старыми рантаймами можно снизить target до ES2020 или ES2019, но это уменьшит набор нативных возможностей.

    3. Сборочная конфигурация: tsconfig.build.json

    Назначение: генерация артефактов в dist/, включая декларации типов (.d.ts) и карты деклараций (.d.ts.map) для точных ссылок из документации и IDE.

    Рекомендуемый вариант:

    // tsconfig.build.json
    {
        "extends": "./tsconfig.base.json",
        "compilerOptions": {
            // --------------------- КОМПИЛЯЦИЯ И ВЫВОД -----------------------------
            "outDir": "dist",
            "rootDir": "src",

            // Генерация типов и их карт (обязательна для библиотек).
            "declaration": true,
            "declarationMap": true,

            // Карты исходников JS для отладки.
            "sourceMap": true,

            // Исключение @internal из публичных деклараций.
            "stripInternal": true,

            // Разрешить эмит артефактов.
            "noEmit": false,

            // Составные проекты и инкрементальная компиляция.
            "composite": true,
            "incremental": true,
            "tsBuildInfoFile": ".tsbuildinfo",
        },
        // Область компиляции
        "include": ["src"],
        "exclude": ["dist", "node_modules", "tests"],
    }

    Пояснения и вариации:

    • declarationMap: true критично для навигации из TypeDoc и IDE к исходникам TypeScript.
    • composite: true облегчает интеграцию с проектными ссылками (Project References), если библиотека разбита на пакеты.
    • В моно-репозитории можно вынести tsBuildInfoFile в общий кэш (например, tmp/ts/.tsbuildinfo).

    4. Конфигурация для разработки/IDE: tsconfig.json

    Назначение: комфортная разработка и проверка типов без генерации выходных файлов.

    Рекомендуемый вариант:

    // tsconfig.json
    {
        "extends": "./tsconfig.base.json",
        "compilerOptions": {
            // IDE-режим: не эмитим файлы
            "noEmit": true,

            // Карты исходников для отладки прямо в IDE
            "sourceMap": true,

            // Явно разрешаем импорт .ts/.mts в NodeNext при разработке
            "allowImportingTsExtensions": true,
        },
        "include": ["src", "tests"],
        "exclude": ["dist", "node_modules"],
    }

    Пояснения и вариации:

    • Если вы не храните тесты в tests/, скорректируйте include.
    • При использовании инструментов, исполняющих TypeScript напрямую (например, ts-node), убедитесь, что их конфигурация согласована с module: "NodeNext".

    5. Согласование с package.json и экспортами

    Для ESM-ориентированных библиотек рекомендуется:

    • в package.json указать "type": "module";

    • в поле exports задать явные точки входа, например: 

      {
          "name": "@your-scope/your-library",
          "type": "module",
          "main": "./dist/index.js",
          "types": "./dist/index.d.ts",
          "exports": {
              ".": {
                  "types": "./dist/index.d.ts",
                  "import": "./dist/index.js",
              },
          },
      }

    • убедиться, что tsconfig.build.json генерирует файлы в dist/ по указанным путям.

    Если нужен гибрид ESM/CJS, вопрос выходит за рамки минимальной настройки: потребуется отдельная сборка CJS (например, с дополнительным tsconfig.build.cjs.json или бандлером), а также двоичное дерево exports с условиями "require"/"default".

    6. Алиасы импортов (baseUrl/paths) и их согласование

    • В tsconfig.base.json заданы:

      "baseUrl": ".",
      "paths": {
        "@/*": ["src/*"],
        "tests/*": ["tests/*"]
      }

    • В исходниках используйте:

      import { something } from '@/module/submodule';

    • Если используется бандлер/раннер, убедитесь в поддержке этих алиасов (например, через соответствующий плагин). Для публикации в npm алиасы в рантайме не участвуют, так как компиляция выдаёт относительные пути либо вы используете бандлер.

    7. Интеграция с TypeDoc

    Для корректной навигации из документации в исходники:

    • TypeDoc должен использовать сборочную конфигурацию tsconfig.build.json (это уже настроено в рекомендуемом typedoc.json);
    • в tsconfig.build.json должны быть включены declaration: true и declarationMap: true;
    • опционально задайте displayBasePath: "src" в typedoc.json, чтобы пути в интерфейсе были короче.

    8. Частые варианты настройки и советы

    8.1. Браузерные библиотеки

    • Добавьте DOM-типы:

      "lib": ["ES2022", "DOM"]

    • Если применяется JSX/TSX, укажите:

      "jsx": "react-jsx"

      и соответствующие типы для среды.

    8.2. Более старые рантаймы

    • Снизьте target и lib (например, ES2020) и обеспечьте полифилы на стороне потребителя.

    8.3. Оптимизация проверок

    • skipLibCheck: true ускоряет компиляцию, не проверяя типы зависимостей. Для диагностики проблем в сторонних типах можно временно выключать.

    8.4. Проектные ссылки (монорепозиторий)

    • Включайте composite: true в каждом пакете.
    • В корневом конфиге используйте references на пакеты, чтобы ускорить сборку и навигацию.

    9. Контрольная карта настроек (короткая форма)

    1. База (tsconfig.base.json): target/lib, module: NodeNext, строгие проверки, алиасы @/*.
    2. Сборка (tsconfig.build.json): outDir, rootDir, declaration + declarationMap, sourceMap, stripInternal, composite + incremental.
    3. IDE (tsconfig.json): noEmit, sourceMap, allowImportingTsExtensions.
    4. Exports: синхронизация dist/ с package.json (main, types, exports).
    5. TypeDoc: использовать tsconfig.build.json, обеспечить declarationMap.
    6. Алиасы/раннеры: согласовать paths с инструментами исполнения/сборки.

    11. Проверка и CI

    • Локальная проверка типов: npx tsc -p tsconfig.json --noEmit

    • Сборка артефактов: npx tsc -p tsconfig.build.json

    • Рекомендуется добавить задачи в package.json: 

      {
          "scripts": {
              "typecheck": "tsc -p tsconfig.json --noEmit",
              "build": "tsc -p tsconfig.build.json",
              "clean": "rimraf dist .tsbuildinfo",
          },
      }

    Настройка завершена. Конфигурации взаимно согласованы и подходят для библиотеки с ESM-ориентацией, строгой типизацией и корректной интеграцией с документацией.
